# Handover 2026-07-07

## Context

The project started from a planning discussion about a reusable standard for new SaaS projects. The first idea was a single large master prompt. The better direction is a repository-based standard with vision, constitution, specification, prompts, reviews, decisions, and handovers.

## Main decision

The repository is the source of truth. Chat windows are temporary workspaces.

Do not store the full conversation as canonical documentation. Store distilled context, decisions, risks, and next steps.

## Intended direction

The project should become a model-neutral development manifest for creating new SaaS products from empty repositories.

Tool-specific instructions should be generated as adapters from the canonical standard, not maintained as competing sources of truth.

## Initial scope

The first version focuses on building new SaaS products, not analyzing or modernizing existing projects.

A later second version may define a separate workflow for auditing and improving existing codebases.

## Planned core areas

- project-owned memory
- agent registry
- handover protocol
- decision records
- research protocol
- planning before implementation
- architecture comparison
- review and simplification
- SaaS foundation standard
- provider foundation standard
- security, reliability, testing, documentation, and release standards

## Important warning

Do not turn this into a huge unmaintainable prompt. Build the standard first. The master prompt should reference the standard instead of duplicating all knowledge.
